# Task 5.1 Plan — Reminder Preferences + Web Push Subscription

**Story:** US-TRK-09 (configuration side)
**ADR:** ADR-007 (PWA + Web Push), ADR-011 (Resend email fallback used by Task 5.2)
**Scope this task:** *configuration* surface only. Dispatch happens in Task 5.2.

---

## What ships

1. **Domain validation** (`lib/notifications/domain/validation.ts`)
   - `HH:MM` 24-hour `reminderTime` regex (`/^([01]\d|2[0-3]):[0-5]\d$/`).
   - IANA timezone whitelist via `Intl.supportedValuesOf('timeZone')` (Node ≥ 18) with
     a fallback `Intl.DateTimeFormat(undefined, { timeZone }).format()` validator.
   - `ChannelPreference` literal union: `'PUSH' | 'EMAIL' | 'BOTH'`.
   - `PushPermissionState` literal union: `'NOT_PROMPTED' | 'GRANTED' | 'DENIED'`.

2. **Application service** (`lib/notifications/application/ReminderService.ts`)
   - `getReminderPreference(userId)` — returns existing row or `null` (no implicit create).
   - `upsertReminderPreference(userId, { reminderTime, timezone, channel, enabled, emailFallbackEnabled })`
     — wrapped in `withAudit`; emits `REMINDER_PREFERENCE_UPDATED`.
   - `setPushPermissionState(userId, state)` — separate audit (`PUSH_PERMISSION_STATE_CHANGED`).
   - `registerPushSubscription(userId, { endpoint, p256dh, auth })` — upserts on `endpoint`
     uniqueness; wrapped in `withAudit`; emits `PUSH_SUBSCRIPTION_REGISTERED`.
   - `removePushSubscription(userId, endpoint)` — userId-scoped delete; `withAudit` →
     `PUSH_SUBSCRIPTION_REMOVED`. Returns silently if no row.

3. **Server actions** (`app/actions/notifications/`)
   - `update-reminder-preferences.ts` — Zod-validated form submit.
   - `register-push-subscription.ts` — accepts the browser-supplied `PushSubscription.toJSON()` payload, persists.
   - `remove-push-subscription.ts` — unsubscribe.
   - `set-push-permission-state.ts` — written on `Notification.permission` change.

4. **UI** — new `Reminders` section at `/settings`:
   - `_components/RemindersForm.tsx` (client) — controlled inputs for time, timezone,
     channel, enabled, email-fallback.
   - `_components/PushSubscriptionPanel.tsx` (client) — handles browser permission flow,
     registers/unregisters subscription via service worker, surfaces iOS install-prompt UX.
   - Settings RSC page (`page.tsx`) loads the user's existing preference (if any) and
     renders both subsections.

5. **Service worker** (`public/sw.js`)
   - Add `push` and `notificationclick` event listeners. Notification body: dose summary
     (passed in by Task 5.2's payload). Click → `clients.openWindow('/tracker')`.

6. **VAPID env** (`.env.example`)
   - `WEB_PUSH_PUBLIC_KEY=""` (base64url, ~88 chars)
   - `WEB_PUSH_PRIVATE_KEY=""` (base64url, ~43 chars)
   - Surface the public key to the client via a `/api/notifications/vapid-public-key`
     GET endpoint (no secret leak, but easier than a runtime config import).

7. **Audit actions** — add to `lib/audit/domain/AuditEvent.ts`:
   - `REMINDER_PREFERENCE_UPDATED`
   - `PUSH_PERMISSION_STATE_CHANGED`
   - `PUSH_SUBSCRIPTION_REGISTERED`
   - `PUSH_SUBSCRIPTION_REMOVED`
   - Category: `'Auth'` (no `Notification` category — keep AuditCategory closed for now;
     these are user-account settings).

## What does NOT ship in this task

- **Reminder dispatch cron** — Task 5.2.
- **Server-side push send** — Task 5.2 (`web-push` dep added there).
- **Background email fallback when push fails** — Task 5.2.
- **Snooze / acknowledge from notification** — out of scope (US-TRK-09 covers config; ack happens at /tracker).

- AC-6: `upsertReminderPreference` rejects unsupported channel values.

  re-registration does not duplicate, but reassigns to the same user.
- AC-8: `registerPushSubscription` is userId-scoped — endpoint owned by another user
  cannot be reassigned (P2002 surface; we treat the endpoint as the device's identity
  and reject reassignment cross-user to prevent push-hijack).
- AC-9: `removePushSubscription` deletes only the actor's row (userId predicate).
- AC-10: `setPushPermissionState` validates the literal union; writes audit.

Action-layer tests:
- AC-A1: `updateReminderPreferencesAction` requires session; throws `unauthorized`.
- AC-A2: `registerPushSubscriptionAction` requires session.
- AC-A3: Each action calls the underlying service with `session.user.id` (proves
  IDOR-safe).

Validation tests are colocated `lib/notifications/domain/validation.test.ts` with

## Identity scoping audit

| Query | Scope |
|-------|-------|
| `reminderPreference.findUnique({ where: { userId } })` | ✅ userId in predicate |
| `reminderPreference.upsert({ where: { userId } })` | ✅ unique on userId |
| `pushSubscription.findFirst({ where: { endpoint } })` (pre-upsert hijack check) | ⚠️ system-wide endpoint lookup — but returns only `userId` field; rejected if `existing.userId !== actor.userId`. Equivalent to AuthRepository's email lookup pattern. Document as approved boundary in `AGENTS.md`. |
| `pushSubscription.create / update` | ✅ ID-scoped after the hijack check |
| `pushSubscription.deleteMany({ where: { userId, endpoint } })` | ✅ userId in predicate |

The endpoint-uniqueness lookup is the only non-userId-scoped query. Treat it as
"endpoint is a globally-unique device identifier; we must check ownership before
overwriting." Document the exception in AGENTS.md before pushing — same workflow as
`markOrdersStale` / `processPendingDeletions`.

## Out-of-band notes

- VAPID keys are not present in any CI env — the public-key route gracefully
  returns 503 when unset (so dev environments don't crash). The
  `PushSubscriptionPanel` reads the key via fetch on mount; if 503, it shows the
  email-only fallback UI.
- iOS Safari: when `'standalone' in window.navigator && navigator.standalone === false`
  AND user-agent contains `iPhone`/`iPad`, the panel shows the "Add to Home Screen"
  instructions instead of the permission button.
