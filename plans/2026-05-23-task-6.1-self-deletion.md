# Task 6.1 Plan — Account Self-Deletion

**Story:** US-AUT-02 (delete + 48h-delay + immediate + cancel)
**Reuses:** `generateUserDataExport` (Task 6.2), `AccountDeletionRequest`
schema (Task 4.3 — `requestedByUserId` nullable for self-deletion),
`processPendingDeletions` cron (Task 4.3 — extend with self-delete branch).

---

## What ships

1. **Service** — `lib/auth/application/scheduleAccountDeletion.ts`
   - `requestSelfDeletion(userId, { confirmEmail })` → schedules a 48h-
     delayed deletion. Typed-email gate (case-insensitive trim) matches
     Task 4.3's pattern.
     - Generate the full export and **email it synchronously** BEFORE
       any deletion side-effect (Task 4.3 lesson: export-before-
       destruction is the cardinal rule).
     - Revoke any Telegram session.
     - Create `AccountDeletionRequest` with `requestedByUserId: null` and
       `scheduledFor: now + 48h`.
     - Set `User.status = 'DELETION_PENDING'`.
     - Audit: `ACCOUNT_DELETION_SCHEDULED`.
   - `cancelSelfDeletion(userId)` → restores during the 48h window.
     - Delete the ADR scoped to `{ userId, status: 'PENDING' }`.
     - Set `User.status = 'ACTIVE'` (was DELETION_PENDING).
     - Audit: `ACCOUNT_DELETION_CANCELLED`.
   - `requestImmediateDeletion(userId, { confirmEmail, acknowledged })`
     → executes the full delete now. Double-confirm = typed email AND
     `acknowledged: true`. Reuse the cron's destructive sequence: pre-
     delete user's orders, then `user.delete`. Audit: `ACCOUNT_DELETED`
     with `mode: 'immediate_self'` metadata.

2. **Cron extension** — `processPendingDeletions`
   - Branch on `req.requestedByUserId === null && user.managedBy === null`
     as the approved self-deletion case. Same destructive sequence as
     the managed-user path, but with audit `mode: 'delayed_self'`.
   - The existing "stale" branch (mismatched requestor) still fires
     for the managed-user case.

3. **Server actions** — `app/actions/account/`
   - `schedule-deletion.ts` — typed-email confirmation.
   - `delete-immediately.ts` — typed email + `acknowledged: true`.
   - `cancel-deletion.ts` — no input besides session.

4. **UI** — `/settings`
   - "Danger zone" section with destructive styling.
   - Two-step modal: choose 48h-delay vs immediate.
   - 48h-delay path opens typed-email confirmation.
   - Immediate path opens BOTH typed-email AND a second
     "I-understand-this-is-irreversible" checkbox.
   - Cancel banner: shown when user.status === 'DELETION_PENDING',
     surfaces a single "Cancel deletion" button.

5. **Audit actions** — no new actions; reuse the three from Task 4.3.

## Identity-scoping

No new exception. The self-deletion service operates within the
authenticated session — every DB write is `userId`-scoped to the actor.
The cron extension shares the same global-scan exception that's already
documented for `processPendingDeletions`.

## TDD: acceptance tests

`tests/acceptance/AUT-self-deletion.test.ts`:

- AC-1: schedule with mismatched typed email → throws `email_mismatch`,
  no side-effects.
- AC-2: schedule with correct email but missing export delivery →
  throws `export_email_failed` and ADR is NOT created.
- AC-3: schedule success — export sent, Telegram session revoked, ADR
  created with scheduledFor +48h and requestedByUserId=null, User.status
  becomes DELETION_PENDING, audit emitted.
- AC-4: cancel during pending window → ADR deleted, User.status restored
  to ACTIVE, audit emitted.
- AC-5: cancel when no pending ADR → throws `no_pending_deletion`.
- AC-6: immediate deletion with typed-email mismatch → throws.
- AC-7: immediate deletion with acknowledged=false → throws
  `acknowledgment_required`.
- AC-8: immediate deletion success — user row deleted, ACCOUNT_DELETED
  audit emitted with `mode: 'immediate_self'`, Telegram session revoked.

`tests/acceptance/ADM-self-deletion-cron.test.ts` (extends existing cron
test):

- AC-9: cron with self-deletion ADR (requestedByUserId=null,
  user.managedBy=null) → deletes user, writes ACCOUNT_DELETED with
  `mode: 'delayed_self'`.
- AC-10: cron with self-deletion ADR but user.managedBy is set →
  treats as stale (paranoid invariant: a managed user's self-deletion
  was once issued; if their Power User has been migrated since, abort).

## Out-of-scope

- R2 / signed-URL handoff for >17MB exports (Task 6.2 follow-up).
- Reactivation flow for "I accidentally deleted my account" — once the
  cron runs, the row is gone. ADR-009 audit history survives.
