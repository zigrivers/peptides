# Task 6.3 Plan — Audit Log Purge Cron + Backup Verify Cron

**ADRs:** ADR-009 (90-day audit retention), ADR-012 (Railway Cron), PRD §8.7 (backup monitoring)

---

## What ships

1. **Audit purge** — `lib/audit/application/AuditPurgeService.ts`
   - `purgeOldAuditEvents(now: Date, retentionDays = 90)`: deletes
     `AuditEvent` rows whose `timestamp < now - retentionDays`. Returns
     `{ deleted: number, cutoff: Date }`.
   - Idempotent: safe to manually re-run; each invocation simply
     removes whatever's currently past the cutoff.
   - Approved exception: this is a system-level cron with no session
     context. The findMany/deleteMany ARE global by design.

2. **Audit purge cron** — `app/api/cron/audit-purge/route.ts`
   - `POST` with `Authorization: Bearer ${CRON_SECRET}`. Daily 04:00 UTC.
   - Returns the deleted count.

3. **Backup verify cron** — `app/api/cron/backup-verify/route.ts`
   - `POST` with `Authorization: Bearer ${CRON_SECRET}`. Daily 05:00 UTC.
   - Implementation note: Railway Cron's existence proves the platform
     is healthy; per PRD §8.7 the operational guarantee is "alert via
     Sentry on miss". For v1 we emit a structured `console.info` log
     and return 200 with `{ ok: true, verifiedAt }`. A future Sentry
     check-in heartbeat will replace the console.info — the route's
     job is to BE the heartbeat. Doc the upgrade path inline.

4. **AGENTS.md / CLAUDE.md exception** for `purgeOldAuditEvents` (same
   shape as `markOrdersStale`, `processPendingDeletions`).

## Not in scope

- Sentry integration (PRD lists it but no SENTRY_DSN is wired up yet).
- Restoring purged events (audit is append-only; purging is final).
- Per-category retention overrides (ADR-009 is a flat 90 days).
