# Task 5.2 Plan — Reminder Dispatch Cron (15-minute tick)

**Story:** US-TRK-09 (dispatch side)
**ADRs:** ADR-012 (Railway Cron, 15-min cadence), ADR-007 (Web Push), ADR-011 (Resend email fallback)
**Endpoint:** `POST /api/cron/dose-reminders` — `Authorization: Bearer ${CRON_SECRET}`

---

## What ships

1. **Schema migration**
   - Add nullable `lastDispatchedAt: DateTime?` to `ReminderPreference` so the
     dispatcher can dedupe within the same local calendar day across the
     four cron ticks that overlap a 15-minute window edge.
   - Migration file: `prisma/migrations/20260523000002_reminder_last_dispatched_at/migration.sql`.

2. **Web push transport** — `lib/notifications/infrastructure/webPush.ts`
   - Lazy `Proxy` around the `web-push` client (mirrors `lib/shared/email.ts`
     pattern — the Resend lesson from Task 4.3). Module import is side-effect
     free; VAPID keys are read at first send.
   - Returns `{ ok: true }` on success, `{ ok: false, expired: boolean, error: string }`
     on failure. `expired = true` for 404 / 410 (subscription gone — caller
     prunes the row).

3. **Reminder email** — `lib/notifications/infrastructure/reminderEmail.ts`
   - `sendReminderEmail(email, displayName | null)`. Subject:
     "Daily peptide reminder". Body: deliberately PII-free per operations §7
     — no compound names, no dose values, no protocol details. Single CTA
     linking to `/tracker`.

4. **Dispatcher** — `lib/notifications/application/ReminderDispatcher.ts`
   - `dispatchDoseReminders(now: Date): Promise<DispatchSummary>`.
   - Algorithm:
     1. `findMany` `ReminderPreference` where `enabled = true`. Approved
        global scan — system-level cron — add AGENTS.md / CLAUDE.md
        exception. Mirror `markOrdersStale`.
     2. For each preference:
        - Compute local now in `pref.timezone` via Intl.DateTimeFormat
          formatToParts; extract `HH:MM` and `YYYY-MM-DD`.
        - Window check: `currentLocalMinutes ∈ [prefMinutes, prefMinutes + 15)`
          mod 1440.
        - Dedupe: `lastDispatchedAt` (formatted in the user's tz)
          ≠ current local date.
        - Continue if both pass.
     3. Compute "due today" — if `getReminderDoseSummary(userId, now)` returns
        zero scheduled doses, skip (don't audit, don't dispatch — there's
        nothing to remind about). `getReminderDoseSummary` reuses
        `listProtocolsForUser` + `isScheduledOn`.
     4. Dispatch:
        - If `channel === 'PUSH'` or `'BOTH'`: send to each `PushSubscription`.
          On `410 Gone` / `404`, `deleteByEndpoint`. If all push sends fail
          AND `emailFallbackEnabled` AND user has an email, also send email.
        - If `channel === 'EMAIL'` or `'BOTH'`: send email.
        - On any send failure, increment `errors` but DO NOT throw — the
          loop must keep going (one user's send failure must not block
          another's). Log via `console.error` (no PII).
     5. After dispatch: update `lastDispatchedAt = now` via `withAudit`,
        emitting `REMINDER_DISPATCHED` audit. Each user's update + audit is
        its own `$transaction` so a write failure doesn't roll back prior
        users' updates.

5. **Cron route** — `app/api/cron/dose-reminders/route.ts`
   - `POST`, `Authorization: Bearer ${CRON_SECRET}`.
   - Calls `dispatchDoseReminders(new Date())`.
   - Returns `NextResponse.json(summary)`.

6. **Audit action** — add `REMINDER_DISPATCHED` to `AuditEvent.ts` in the
   `Notification` category. `actorUserId: 'SYSTEM'`, `subjectUserId: userId`,
   metadata: `{ pushDelivered, emailDelivered, errors }` — no PII.

7. **AGENTS.md / CLAUDE.md exceptions**
   - `dispatchDoseReminders` global `findMany` on `ReminderPreference` — same
     pattern as `markOrdersStale` and `processPendingDeletions`.

8. **Dependency** — add `web-push@^3.6.7` to `package.json`. Server-only.

## TDD: tests

`tests/acceptance/NTF-reminder-dispatch.test.ts`:

- AC-1: empty preferences → returns `{examined: 0, dispatched: 0}`.
- AC-2: skips preferences with `enabled = false`.
- AC-3: skips preferences whose local current time is outside the 15-min window.
- AC-4: in-window + no `lastDispatchedAt` → dispatches; updates `lastDispatchedAt`.
- AC-5: in-window + `lastDispatchedAt` is earlier-today-local → skipped (dedupe).
- AC-6: in-window + `lastDispatchedAt` is yesterday-local → dispatches.
- AC-7: zero scheduled doses today → skipped, no audit.
- AC-8: channel=PUSH + push success → emails NOT sent. Audit metadata reflects.
- AC-9: channel=EMAIL → push not attempted; only email sent.
- AC-10: channel=BOTH → both sent.
- AC-11: all push subscriptions expired (410) → rows removed and email
  fallback fires when `emailFallbackEnabled`.
- AC-12: push send transient failure → does NOT prune subscription, but no
  fallback email unless `channel === 'BOTH'` or `emailFallbackEnabled`.
- AC-13: dispatcher continues after a per-user exception.
- AC-14: writes `REMINDER_DISPATCHED` audit on success.

`tests/acceptance/NTF-cron-endpoint.test.ts`:

- AC-A1: 401 without bearer.
- AC-A2: 401 with bad bearer.
- AC-A3: 200 with correct bearer + invokes dispatcher.

Edge cases covered by smaller unit tests (`lib/notifications/domain/window.test.ts`):
- Wrap-around: pref 23:50, current 00:01 → out of window.
- Wrap-around: pref 00:00, current 00:14 → in window.
- Exact boundary: pref 07:00, current 07:14:59 → in. 07:15:00 → out.

## Identity scoping

Cron entry: `findMany ReminderPreference { enabled: true }`. Approved
exception (global system-level scan). All subsequent reads (`PushSubscription`
list, `Protocol` list) are scoped by `userId` in their existing repos.

## Out-of-scope

- Snooze / "remind me later" — not in US-TRK-09 v1.
- Push payload encryption beyond what `web-push` already does (it handles VAPID +
  AES-GCM internally).
- Per-protocol granular reminders. The reminder is per-user-daily.
- Resend retries — single attempt; failure is logged but not re-queued.
