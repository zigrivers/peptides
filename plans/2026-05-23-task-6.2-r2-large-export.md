# Task 6.2 R2 Upgrade — Large-Export Pipeline + Cleanup Cron

**Story:** US-AUT-02 export side (large-payload branch)
**ADRs:** ADR-014 (R2 object storage), ADR-012 (export-cleanup cron daily 03:00 UTC)

## What ships

1. **Lazy R2 client** — `lib/shared/r2.ts`
   - Wraps `@aws-sdk/client-s3` + `@aws-sdk/s3-request-presigner` in a
     lazy-init pattern (same shape as `lib/shared/email.ts` Resend
     wrapper). Module import is side-effect free; the SDK is only
     instantiated when R2 env vars are present and the client is first
     used.
   - Surfaces a small interface: `putExport`, `presignExportDownload`,
     `deleteExport`, `listExportsOlderThan`.
2. **Export storage helper** — `lib/auth/infrastructure/exportStorage.ts`
   - Composes the R2 client into the key-namespacing the rest of the
     app expects: `exports/{userId}/{uuid}.json`.
   - Each helper has a `R2_CONFIGURED` sentinel — when env is unset, the
     helpers throw `r2_not_configured`. The export pipeline catches
     this and falls back to "export_too_large" so behaviour matches
     Task 6.2 v1 when R2 isn't wired.
3. **Updated `requestDataExport`**:
   - Below the inline threshold → unchanged (email attachment).
   - Above the inline threshold:
     - Upload buffer to R2 under `exports/{userId}/{requestId}.json`.
     - Presign a 7-day GET URL.
     - Persist `downloadUrl` + `expiresAt` (now + 7d) on the
       `DataExportRequest` row.
     - Send a "your export is ready" email with the link (no
       attachment).
     - Audit DATA_EXPORT_DELIVERED with
       `delivery: 'r2-link', bytes, objectKey`.
4. **Cleanup cron** — `app/api/cron/export-cleanup/route.ts`
   - Daily 03:00 UTC; `Authorization: Bearer ${CRON_SECRET}`.
   - Lists R2 objects older than 7 days under the `exports/` prefix,
     deletes each, and nulls out the corresponding
     `DataExportRequest.downloadUrl` rows.
   - Idempotent. Returns `{ deletedObjects, expiredRequestRows }`.
5. **AGENTS.md / CLAUDE.md exception** for the cleanup cron's global
   scan of `DataExportRequest` (system-level cron, no userId predicate
   on the listObjectsV2; the corresponding DB update IS userId-scoped
   via the row's own userId).

## Identity-scoping

- `presignDownload` returns a URL that grants object-level access — no
  Prisma identity surface. The key is derived from `userId` + request
  UUID, both server-side.
- The cleanup cron's R2 list is global by design (one bucket prefix
  for all users). The corresponding DB updates use `updateMany WHERE
  downloadUrl = key` — keys embed userId, so DB rows are implicitly
  scoped, but we still pass userId in the predicate when extractable.

## What does NOT change

- The inline path stays at 17 MB. Codex's recurring false-positive
  about Resend limits is documented in AGENTS.md; the threshold is
  correct.
- Admin-triggered managed-user deletion exports (Task 4.3) currently
  abort on `export_too_large`. Reuse the new R2 path inside
  `AdminService.requestManagedUserDeletion` so a Power User with a
  large managed-user history can still delete them.
