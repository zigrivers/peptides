# Plan: Task 4.4 — Ordering Module Isolation Feature Flag (US-ORD-08)

## Goal
Add a single `DISABLE_ORDERING` environment variable that, when `"true"`, makes the entire ordering bounded context inaccessible: all `/ordering/*` routes return 404, all ordering server actions reject, the dashboard's stale-orders banner and the ordering Telegram-setup link disappear from the UI, and the stale-orders cron is a no-op. Tracker, Reference, Reconstitution, and Admin pillars remain fully functional.

## Stories
US-ORD-08 — Ordering Module Isolation (Non-Functional).
ADR-015 — Bounded Context Isolation for Ordering.

## Non-goals
- **No schema changes.** The flag is environment-only; no Prisma migration.
- **No deletion of ordering code.** Module remains in the repo, just gated at runtime.
- **No per-user RBAC.** The flag is a global toggle controlled by the deploy environment, not a user-level permission.
- **No Playwright E2E.** Repo does not yet have a working authenticated Playwright fixture (the existing `tests/e2e/onboarding.spec.ts` is intentionally skipped per `tasks/lessons.md`). E2E coverage for this flag deferred to the future E2E testing wave; functional coverage lands as Vitest acceptance tests.

## Current System Summary
- Ordering UI pages live under `app/(dashboard)/ordering/`:
  - `page.tsx` (vendor list)
  - `orders/page.tsx` (order history)
  - `new/page.tsx` (new vendor)
  - `[id]/page.tsx` (vendor detail)
  - `_components/{VendorStatusBadge,OrderStatusBadge}.tsx`
- Ordering server actions live in `app/actions/ordering/{telegram-auth,vendor-product,vendor}.ts`.
- Ordering domain code lives in `lib/ordering/` (`application/`, `domain/`, `infrastructure/`).
- Ordering-adjacent UI surfaces:
  - `app/(dashboard)/dashboard/page.tsx` imports `getStaleOrderCount` from `@/lib/ordering/application/OrderService` and renders a stale-orders banner (line ~42) linking to `/ordering/orders`.
  - `app/(dashboard)/settings/telegram/page.tsx` is the MTProto auth UI — this is ordering-only because Telegram is the order-send transport.
- Ordering cron: `app/api/cron/stale-orders/route.ts` (POST, CRON_SECRET-gated).
- Middleware (`middleware.ts`) already runs on all routes except `/api/auth`, `/api/cron`, and static assets.

## Proposed Architecture

**Single source of truth**: `lib/shared/featureFlags.ts` exports `isOrderingDisabled(): boolean` that reads `process.env.DISABLE_ORDERING === 'true'`. All gates call this helper — no inline `process.env` checks.

**Three layers of enforcement (defense in depth):**

1. **Edge middleware**: `middleware.ts` returns a 404 NextResponse for any pathname starting with `/ordering/` or matching `/ordering` exactly, when the flag is on. This guarantees the routes are inaccessible before any RSC code runs. Pages get static 404 behavior without per-page boilerplate.

2. **Server-action guard**: each module in `app/actions/ordering/*.ts` exports actions that check the flag at the top and `throw new Error('ordering_disabled')` when set. (Defense-in-depth: server actions are reachable via direct POST regardless of UI; the middleware only blocks navigation routes.)

3. **API/cron guard**: `app/api/cron/stale-orders/route.ts` short-circuits with `200 { skipped: true, reason: 'ordering_disabled' }` when the flag is on. (Not a 404 because the cron is invoked by Railway with `CRON_SECRET` — we want a clean no-op, not a noisy failure.)

**UI hiding:**
- Dashboard stale-orders banner: gate its rendering and the `getStaleOrderCount` call behind `isOrderingDisabled()` so the page doesn't import ordering domain code when disabled.
- Settings page (if it links to `/settings/telegram`): hide the Telegram link. Need to verify whether such a settings index exists.
- `/settings/telegram` itself goes through the middleware gate as it's the Telegram-auth UI exclusively for ordering. NOTE: this is not under `/ordering` so the middleware rule needs to include it explicitly OR the page calls `notFound()` when disabled. **Decision: have the page call `notFound()` from `next/navigation` when the flag is on, rather than expanding the middleware match.** Keeps the middleware regex tight and locates the policy with the feature owner.

## Step-by-Step Tasks

### Step 1: TDD — Write acceptance test skeleton

Create `tests/acceptance/ORD-isolation.test.ts` with `vi.stubEnv('DISABLE_ORDERING', 'true')` / `vi.unstubAllEnvs()` lifecycle and these describe blocks (start as `it` with implementations, not `it.todo`, since we'll implement next):

- **`isOrderingDisabled()`**:
  - returns `false` when env var is unset
  - returns `false` when env var is `"false"`, `"0"`, `"FALSE"`, anything other than the exact string `"true"`
  - returns `true` only when env var is exactly `"true"`

- **Server action guards** (one block per module — vendor, vendor-product, telegram-auth):
  - each public exported action throws `ordering_disabled` when flag is on
  - each action proceeds normally when flag is off (mock session + happy path)

- **Cron route**:
  - returns `{ skipped: true, reason: 'ordering_disabled' }` with 200 status when flag is on (still requires CRON_SECRET — unauthorized requests still 401)
  - proceeds normally when flag is off

Middleware-level 404 behavior is not unit-testable inside Vitest (would need an integration runner). Document this and rely on the middleware code being short + reviewable.

### Step 2: Implement `lib/shared/featureFlags.ts`

```ts
export function isOrderingDisabled(): boolean {
  return process.env.DISABLE_ORDERING === 'true';
}
```

Pure function, no module-level state. Tests in step 1 cover this.

### Step 3: Middleware gate

In `middleware.ts`, immediately after the public-path check and before the auth check, add:

```ts
if (isOrderingDisabled() && (pathname === '/ordering' || pathname.startsWith('/ordering/'))) {
  return new NextResponse(null, { status: 404 });
}
```

Place the check before the auth redirect so disabled-ordering returns 404 to both authenticated and unauthenticated clients (no information leak: the route looks like it doesn't exist).

### Step 4: Server-action guards

For each ordering action module, add a top-of-file guard helper:

```ts
function assertOrderingEnabled() {
  if (isOrderingDisabled()) throw new Error('ordering_disabled');
}
```

Call `assertOrderingEnabled()` as the first line of every exported action.

### Step 5: Cron route guard

In `app/api/cron/stale-orders/route.ts`, after the CRON_SECRET check, return early:

```ts
if (isOrderingDisabled()) {
  return NextResponse.json({ skipped: true, reason: 'ordering_disabled' });
}
```

### Step 6: UI gates

- `app/(dashboard)/dashboard/page.tsx`: wrap the stale-orders banner block (and the `getStaleOrderCount(userId)` call that feeds it) in `!isOrderingDisabled()`. The import statement stays (no cost), but the runtime call is skipped.
- `app/(dashboard)/settings/telegram/page.tsx`: at the top of the default export, `if (isOrderingDisabled()) notFound();`. Import `notFound` from `next/navigation`.

### Step 7: Documentation

- Add a `### DISABLE_ORDERING` subsection to `docs/operations-runbook.md` §1.2 describing the variable, valid values, and the regulatory-compliance use case from ADR-015.
- If `.env.example` exists, add `DISABLE_ORDERING=false` with a `# Set to "true" to disable the ordering module per ADR-015` comment. If it does not exist, skip (do not create new env-template files speculatively).

### Step 8: Verify

Run `pnpm check`. All 497+ existing tests plus new tests must pass.

## Acceptance Criteria

- **AC 1 (Isolation)**: With `DISABLE_ORDERING=true`, `GET /ordering`, `GET /ordering/orders`, `GET /ordering/new`, `GET /ordering/<any-id>`, and `GET /settings/telegram` all return 404.
- **AC 2 (UI)**: Dashboard does not render the stale-orders banner. Settings/telegram page is not reachable.
- **AC 3 (Integrity)**: With the flag on, `pnpm dev` boots cleanly, `/dashboard`, `/tracker`, `/reference`, `/reconstitution`, and `/admin` all render and function. Tracker dose-logging, reference catalog browsing, reconstitution calculator, and admin panel are unaffected.
- **AC 4 (Server actions)**: Direct invocation of any `app/actions/ordering/*` action throws `ordering_disabled` when flag is on.
- **AC 5 (Cron)**: `POST /api/cron/stale-orders` with valid CRON_SECRET returns 200 + `{ skipped: true, reason: 'ordering_disabled' }` when flag is on; without CRON_SECRET returns 401 as before.
- **AC 6 (Flag off)**: With `DISABLE_ORDERING` unset or any value other than the literal string `"true"`, everything works as it does today on `main`.

## Risks and Edge Cases

- **Env-var string match**: `DISABLE_ORDERING === 'true'` matches only the exact string. Common mistakes (`True`, `TRUE`, `1`, `yes`) silently leave ordering ENABLED. This is the conservative default — explicit opt-in for a destructive switch — but document it clearly in operations-runbook.
- **Server-action bypass risk**: The middleware only gates page routes. A future maintainer adding a new ordering action must also add `assertOrderingEnabled()` to it. Mitigation: the test file enumerates every action module and asserts the guard; CI will flag missing coverage when a new action is added.
- **Telegram session leftover**: With the flag flipped on, any user who has an existing `TelegramSession` row keeps it. The session is not invalidated. This is intentional — the flag is a routing/UI gate, not a data-destruction operation. Flipping the flag off later restores the user's existing session. Document this in operations-runbook.
- **Dashboard import**: The `getStaleOrderCount` import in `dashboard/page.tsx` still pulls in ordering domain code at build time even when the flag is on. Lazy-loading inside the conditional would shave a few KB. Not worth the complexity for an env-flag-gated path; flagged as future cleanup.
- **Middleware ordering**: The 404 must fire BEFORE the unauthenticated redirect, otherwise an anonymous request to `/ordering` redirects to `/login?callbackUrl=/ordering` and leaks that the path exists. Test by inspecting middleware code.
